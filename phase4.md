1-1. 장비 구조

유닛당 장비 슬롯은 3개로 고정한다.

무기 1
방어구 1

이 구조가 좋은 이유는 단순하다.
모바일 UI가 덜 복잡하고, 장비 선택이 명확하다.
수집은 많아도 실제 고민은 “누구에게 무엇을 끼울 것인가”로 압축된다.

1-3. 장비 수치 공식

전투 계산은 너무 복잡하게 시작하지 않는 편이 좋다.
아래 정도면 구현이 깔끔하다.

final_atk = unit.base_atk + weapon.atk + flat_bonuses
final_def = unit.base_def + armor.def + flat_bonuses
final_res = unit.base_res + armor.res + flat_bonuses
final_hit = unit.base_hit + weapon.hit + flat_bonuses
final_avo = unit.base_avo + armor.avo + flat_bonuses
final_crit = unit.base_crit + weapon.crit + flat_bonuses
range_min/max = weapon.range_min/max

패시브 특성은 마지막에 계산한다.

4장 기억 조각 연출

바실이 쓰러지면 리안은 성물에 손을 대고 짧은 기억을 본다.

차가운 기록실.
물소리가 울리는 돌벽.
흰 의복을 입은 사람들이 “정화”를 말하지만, 실상은 기억을 분리하고 세척하는 절차를 수행하고 있다.

그리고 낯익은 목소리 하나.

“공백은 자연현상이 아니다. 관리 가능한 현상이다.”

리안은 그 말을 듣고 직감한다.

3-3. 비교 패널

아이템을 탭하면 현재 장착 아이템과 비교하는 패널이 바로 떠야 한다.

예시:

명중  81 → 76   (-5)
저항  11 → 11   (0)
효과: 치유 시 대상 RES +1

능력치만 보여주면 안 되고,
현재 맵/직업에 체감되는 효과가 한 줄 요약으로 같이 보여야 한다.

예시:

“세린에게 적합: 치유 보조 강화”
“티아에게 적합: 숲맵에서 효율 상승”

We are adding chapter 4 content and a camp equipment UI to a Godot 4 mobile-first tactics RPG.

Constraints:
- GDScript only
- Keep UI tap-driven, no drag and drop
- Randomness applies only to boss-dropped weapons
- Armor and accessories are fixed data
- Equipment calculations must live in services, not UI scenes
- Keep the project runnable after each task
- No unrelated refactors

Task:
Implement phase 1 of the camp equipment flow:
- CampHub scene
- PartyLoadout scene
- weapon / armor / accessory slots
- equip / unequip flow
- filtered inventory bottom sheet
- stat comparison preview
- persistent save/load for equipped items

Acceptance criteria:
- Only compatible items can be equipped
- Stats update immediately in preview and in battle
- Equipped items persist across scene changes and app reload
- New files and changed files are summarized at the end
- Remaining TODOs are listed separately

여기까지 붙이면 구조가 상당히 단단해진다.
이제 캠페인은 스토리 전투 + 숨은 수집 + 보스 파밍 + 장비 성장이 한 고리로 돌아간다.
