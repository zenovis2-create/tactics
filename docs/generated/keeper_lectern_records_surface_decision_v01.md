# Keeper Lectern Records Surface Decision V01

## Decision

For `selective expansion pass 2`, the preferred `keeper_lectern` destination is:

- a records-side support surface adjacent to `evidence`

Not:

- a new first-wave dossier category
- a briefing-first surface
- a broad archive-wide panel role

## Why This Is The Best Current Choice

The current `Records` section already exposes:

- memory
- evidence
- letters

That means the safest next move is not to multiply records categories blindly.

The safest next move is:

- add support next to the existing records presentation

This keeps:

- `evidence` as proof and witness authority
- `keeper_lectern` as archive handling and mediated access

## Preferred Form

Preferred form:

- a support block or adjacent records note

This should sit beside the current `evidence` surface rather than replacing it.

It should function as:

- archive handling context
- keeper guidance context
- controlled-access context

It should not function as:

- the main evidence surface
- a new canonical proof category

## Why A Separate Full Section Is Not Preferred Yet

A separate records category is not preferred in the current slice because:

1. the records section is still compact
2. `keeper_lectern` is not yet broad enough to justify a full standalone lane
3. the risk of confusing it with `evidence` is still too high if the UI weight
   is too similar

## Current Recommendation

If implementation happens later, do it in this order:

1. add a support block adjacent to `evidence`
2. verify that `keeper_lectern` still reads as archive handling
3. only later consider a stronger standalone records role

## Working Conclusion

`keeper_lectern` should expand next to the current records presentation, not
above it and not instead of it.
