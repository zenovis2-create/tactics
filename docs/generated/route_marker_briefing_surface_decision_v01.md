# Route Marker Briefing Surface Decision V01

## Decision

For `selective expansion pass 2`, the preferred `route_marker` destination is:

- conditional tactical-note support

Not:

- first-wave briefing ownership
- codex-first destination
- generic navigation surface

## Why This Is The Best Current Choice

`route_marker` is already live as:

- a stage object family
- a route-reading family

Its value is real, but still narrower than:

- `battery`
- `floodgate`
- `chain_control`

That means it should support briefing only when route commitment is central,
not when the stage is merely using a sign-like object.

## Preferred Form

Preferred form:

- a tactical-note or support-note style briefing role

This should function as:

- route commitment warning
- split-path reading aid
- clarification that one path will narrow or collapse

It should not function as:

- the main preview card
- the main mission objective strip
- a general chapter navigation symbol

## Why It Should Not Take A Larger Role Yet

The family still carries higher drift risk than the first-wave briefing set.

If expanded too aggressively, it can collapse into:

- `gate_control`
- `latch`
- generic signpost flavor

That makes restrained support use the right current choice.

## Current Recommendation

If `route_marker` expands next, do it in this order:

1. allow one conditional tactical-note support use
2. verify that the route-reading contract remains distinct
3. only later consider stronger briefing weight

## Working Conclusion

`route_marker` should enter briefing space as support, not as lead.
