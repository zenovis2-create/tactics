# Selective Live Family Usage Review V01

## Purpose

This document reviews the five live families that are runtime-valid but are not
currently first-wave briefing or first-wave codex-dossier priorities.

Families under review:

- `well`
- `shrine`
- `keeper_lectern`
- `route_marker`
- `latch`

The goal is to decide:

- where each family can safely expand
- where each family should stop

## Review Outcome

### 1. `well`

Safe expansion:

- keep as live stage object family
- allow selective records or codex mention when investigation framing matters

Do not overexpand:

- do not push into standard briefing-first surfaces
- do not treat as a generic sacred-object replacement

Working judgment:

- `stable`
- `selective codex support only`

### 2. `shrine`

Safe expansion:

- keep as live stage object family
- allow chapter-specific codex or explainer support if CH03 ritual logic needs
  reinforcement

Do not overexpand:

- do not promote to universal dossier-first family
- do not treat as interchangeable with `altar`

Working judgment:

- `stable`
- `chapter-specific support only`

### 3. `keeper_lectern`

Safe expansion:

- keep as live stage object family
- allow conditional records/codex usage tied to archive handling and keeper
  mediation

Do not overexpand:

- do not collapse it into `evidence`
- do not put it into briefing-first surfaces by default

Working judgment:

- `stable`
- `records-facing support allowed`

### 4. `route_marker`

Safe expansion:

- keep as live stage object family
- allow conditional briefing support only when route commitment is central to
  planning

Do not overexpand:

- do not promote to codex-first family
- do not let it drift into generic signpost or route machinery semantics

Working judgment:

- `stable`
- `conditional briefing support only`

### 5. `latch`

Safe expansion:

- keep as live stage object family
- allow conditional briefing support when release-state relief is central to the
  encounter

Do not overexpand:

- do not force it into records/codex-first surfaces
- do not treat it as a generic `lever` variant

Working judgment:

- `stable`
- `conditional briefing support only`

## Recommended Usage Matrix

| Family | Keep live stage use | Briefing support | Codex / dossier support | Avoid |
| --- | --- | --- | --- | --- |
| `well` | Yes | No by default | Conditional | generic sacred replacement |
| `shrine` | Yes | No by default | Conditional chapter-specific | altar collapse |
| `keeper_lectern` | Yes | No by default | Conditional | evidence collapse |
| `route_marker` | Yes | Conditional | No by default | generic signpost drift |
| `latch` | Yes | Conditional | No by default | lever or gate-control collapse |

## Working Conclusion

These five families are already useful.

The right move is not to force them into first-wave UI categories.

The right move is:

- keep them live
- expand them only where their meaning stays clean
