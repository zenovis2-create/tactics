# Civic Seal Wiring Decision Memo V01

## Purpose

This memo records the current go-or-hold decision for
[city_seal_dais_01](/Volumes/AI/tactics/assets/props/city_seal_dais_01/spec.md)
as a runtime family candidate.

It exists to prevent the project from reopening the same question without a new
reason.

## Candidate

Proposed family:

- `civic_seal`

Primary first target:

- [ch07_05_city_seal.tres](/Volumes/AI/tactics/data/objects/ch07_05_city_seal.tres)

## Current Decision

Decision:

- `Hold, but next in line after the current live set`

This is not a rejection.

It means the family is still viable, but should not be opened automatically
without one more explicit reason to do so.

## Why It Is Not An Immediate Go

### 1. It still overlaps with two live families

`civic_seal` risks collapsing into:

- `bell` if treated as civic ritual pressure
- `evidence` if treated as authority-bearing record proof

That overlap is manageable, but not yet trivial.

### 2. Its strongest value is not mechanics-first

Unlike `latch`, this candidate is stronger on:

- codex value
- dossier value
- authority-facing chapter identity

and weaker on:

- immediate mechanics proof
- narrow interaction-state distinction

That makes it a good candidate, but not an automatic next wiring.

### 3. The live family set is already broad

The project already has ten live runtime families:

- `well`
- `battery`
- `shrine`
- `floodgate`
- `evidence`
- `bell`
- `chain_control`
- `keeper_lectern`
- `route_marker`
- `latch`

At this point, new families should open only when they clearly reduce ambiguity,
not just because they are interesting.

## Why It Is Still A Strong Candidate

`civic_seal` remains stronger than most other hold candidates because:

1. it may still add a legitimacy grammar that no live family owns
2. it has better codex and dossier survivability than many remaining props
3. CH07 already gives it a plausible first-stage anchor

So the decision is not `no`.

The decision is `not yet automatic`.

## Go Conditions

Open `civic_seal` if one of these becomes true:

1. CH07 or CH09A needs an explicit oath-state or legitimacy-state runtime type
2. codex / dossier implementation shows that `city_seal` reads cleanly apart
   from both `bell` and `evidence`
3. a stage runner or scenario contract proves that seal-state is materially
   different from both warning-state and archive-authority-state

## Hold Conditions

Keep it on hold if:

1. it still reads like `bell + evidence` combined
2. no runner demands a separate state id or control grammar
3. the value is still mostly chapter flavor and records framing

## Working Conclusion

`civic_seal` is the strongest remaining non-hold candidate after the current
live set, but it is not yet a forced wiring decision.

The correct posture is:

- keep it ready
- do not forget it
- do not open it until one more concrete contract appears
