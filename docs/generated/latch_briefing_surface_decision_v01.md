# Latch Briefing Surface Decision V01

## Decision

For `selective expansion pass 2`, the preferred `latch` destination is:

- conditional tactical-note support

Not:

- first-wave briefing ownership
- codex-first destination
- generic mechanism support

## Why This Is The Best Current Choice

`latch` is already live as:

- a stage object family
- a release-state control family

Its value is strongest when the player needs to understand:

- that a blocked path can be released
- that a choke will open
- that the encounter pressure changes after that release

That makes it useful in briefing space, but only as a supporting tactical read.

## Preferred Form

Preferred form:

- one tactical-note style support line

This should function as:

- blocked-path relief warning
- choke-release clarification
- reminder that the route changes after interaction

It should not function as:

- the main preview card
- the main mission objective strip
- a generic machinery icon

## Why It Should Not Take A Larger Role Yet

The family still overlaps too easily with:

- `lever`
- `gate_control`

If expanded too aggressively, it stops reading as release-state gating and turns
into a broad “thing that opens something” family.

That is exactly what the live-family split is trying to avoid.

## Current Recommendation

If `latch` expands next, do it in this order:

1. allow one conditional tactical-note support use
2. verify that the release-state read remains distinct from `route_marker`
3. only later consider stronger briefing weight

## Working Conclusion

`latch` should enter briefing space as support, not as lead.

Its role is to clarify relief, not to own the whole planning surface.
