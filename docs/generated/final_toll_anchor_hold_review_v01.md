# Final Toll Anchor Hold Review V01

## Purpose

This document records why
[bell_dais_01](/Volumes/AI/tactics/assets/props/bell_dais_01/spec.md)
should remain on hold as a runtime family candidate.

The goal is to prevent repeated reopening of the same question without new
evidence.

## Candidate Under Review

Proposed role:

- `final_toll_anchor`

Working target:

- [ch10_05_bell_dais.tres](/Volumes/AI/tactics/data/objects/ch10_05_bell_dais.tres)

## Current Live CH10 Families Already Cover Most Of The Space

Live runtime families already active in the endgame slice:

- `bell`
- `chain_control`

These already cover:

- public or ritual pressure
- bell-line state
- terminal route release
- endgame control-object relief

That leaves very little unclaimed runtime territory for `final_toll_anchor`.

## Why It Stays On Hold

### 1. It overlaps too directly with `bell`

If `bell_dais_01` is framed as:

- final ritual pressure
- final bell authority
- public or sacred toll focus

then it collapses into `bell`.

### 2. It overlaps too directly with `chain_control`

If `bell_dais_01` is framed as:

- the object that changes the final approach
- the object that opens or closes the last route
- the object that weakens Karuon's line

then it collapses into `chain_control`.

### 3. CH10 already has a strong two-object grammar

Current CH10 runtime read is already legible as:

- `bell` = pressure and final ritual threat
- `chain_control` = release and route relief

Adding `final_toll_anchor` now risks creating a third family without a third
clear gameplay contract.

## Existing Evidence Is Not Enough

Current CH10 evidence does show that the bell dais matters:

- [ch10_05_stage.tres](/Volumes/AI/tactics/data/stages/ch10_05_stage.tres)
- [lategame_boss_pattern_runner.gd](/Volumes/AI/tactics/scripts/dev/lategame_boss_pattern_runner.gd)

But the evidence does not yet prove that the dais owns a runtime meaning that is
cleanly separate from:

- `bell`
- `chain_control`

So the object is important, but not yet justified as its own reusable family.

## Keep It As

For now, `bell_dais_01` should remain:

- a strong chapter landmark
- a preview and support-surface asset
- a stage-specific object inside CH10 logic

It should not yet become a reusable runtime family.

## Reopen Conditions

Only reopen `final_toll_anchor` if at least one of these becomes true:

1. CH10 or another late-game stage needs a third distinct endgame object grammar
2. the bell dais starts owning a gameplay contract that neither `bell` nor
   `chain_control` can express cleanly
3. a dedicated non-battle UI surface proves the dais carries a stable identity
   outside current CH10 staging

If none of those conditions are met, do not reopen it.

## Working Conclusion

`final_toll_anchor` is not blocked because the art is weak.

It is on hold because the runtime grammar is still redundant.

Until a clearly separate contract appears, `bell_dais_01` should remain
art-first and stage-specific.
